二、容器中如何实现IP hash的能力

问题分析

负载均衡策略都不能用轮询或者权重，要保证同一个电脑每次访问的节点是同一个，不然管理端用不了，实际该服务的pod 172.16.26.2 172.16.27.7 ，使用的是nodeport xxx.xxx.xxx.xxx:31359 访问。可用会话保持（Session Affinity）功能

正确配置方案

1. SessionAffinity 配置

不需要填写Pod的IP，sessionAffinity: ClientIP 是Service级别的配置，Kubernetes会自动处理IP到Pod的映射。

正确的Service配置示例：

在Kubernetes中，Service的externalTrafficPolicy字段有两个选项：Cluster（默认）和Local。

  • Cluster：将外部流量均匀分配到所有节点（包括没有运行Pod的节点），然后通过kube-proxy转发到Pod。这会隐藏客户端的源IP（因为经过了一次转发），并且可能增加网络跳数。
  • Local：只将流量分配到运行有Service Pod的节点，并且直接由该节点上的kube-proxy转发到本地的Pod。这样可以保留客户端的源IP，并且避免额外的网络跳转。

为什么使用externalTrafficPolicy: Local对于IP哈希会话保持更有利？

当使用sessionAffinity: ClientIP时，Service会根据客户端的IP地址进行哈希，将同一个客户端的请求转发到同一个Pod。但是，如果externalTrafficPolicy是Cluster，那么到达Service的流量可能会先到任何一个节点，然后该节点再转发到目标Pod。在这个过程中，客户端的源IP可能会被丢失（因为经过SNAT），尤其是当流量被转发到不同节点上的Pod时。

而设置externalTrafficPolicy: Local可以确保：

  1. 保留客户端的真实IP地址。
  2. 流量只到达运行有Pod的节点，并且直接由该节点处理，不会跨节点转发。

这样，基于客户端IP的会话保持会更加准确，因为Service看到的客户端IP是真实的，而不是某个中间节点的IP。

注意事项

  1. ClientIP获取问题：在云环境或经过NAT时，可能需要配置externalTrafficPolicy: Local来确保获取真实客户端IP
  2. Pod重启问题：如果Pod重启，会话会中断，需要考虑应用层的会话复制或持久化
  3. 测试方法：从同一客户端多次访问，检查日志确认请求总是到达同一个Pod

总结

您只需要在Service中设置sessionAffinity: ClientIP即可，不需要填写Pod的IP地址。Kubernetes会自动基于客户端IP进行哈希计算，将同一客户端的请求路由到同一个Pod。

容器化之细节问题处理篇04

发表于 2025-09-15 分类于 容器化

问题背景：在k8s种，查看一个mysql 容器化的内存（16G）告警了，表示资源不够了。该mysql pod 有三副本，一主两从。有1从得资源内存耗费很低。且该内存已经由8G已经扩容至16G了。使用kubectl top 查看该mysql的 的确是到达了15334Mi ，但是docker stats看到的内存使用量4.477GiB。为什么同一个对象，显示的内存差异如此之大。

核心原因：统计维度不同

docker stats (显示 4.477 GiB)

  • 统计对象： Docker 容器本身（即 cgroup 的内存使用情况）。
  • 统计内容：RSS，即进程占用的内存。这个值更接近应用程序（比如你的 MySQL 进程）真正占用的、未被磁盘缓存缓冲区的内存
  • 特点： 这个值通常会更小，因为它不包含被内核利用为缓存（Page Cache）的内存。

kubectl top (显示 15334 MiB ≈ 15 GiB)

  • 统计来源： 这个数据来自于 Kubernetes 的监控组件 metrics-server，而 metrics-server 的数据又来自节点的 cAdvisor。

  • 统计内容： 它显示的是容器内存工作集（Working Set） 的大小。

  • 工作集（Working Set） 包括：

    内存使用量（memory.usage_in_bytes）：= RSS + 缓存（包括文件缓存、块设备缓存等） + 内核内存等

参数 含义 相当于

  • 测最大IOPS：bs=4k, iodepth=32 (用摩托车车队送小件)
